## Bug 1: JobDetails.jsx — Uses raw JSON instead of enriched job data
**Severity:** Critical  
**File:** `src/pages/JobDetails.jsx`

### Current Issue
- Line 6: Directly imports `mockData.json` and uses `data.projects`
- Bypasses `getEnrichedJob()` utility, missing computed fields like `fitScore`, `department`, `responsibilities`, `requirements`, `perks`, etc.
- Inconsistent with how `JobApply.jsx` and other pages fetch job data

### Fix Plan
1. Remove direct `mockData.json` import
2. Import `getEnrichedJob` from `../data/jobUtils`
3. Replace `const job = JOBS.find(...)` with `const job = getEnrichedJob(jobId)`
4. Keep existing null check (line 13) — it already handles missing jobs correctly
5. The enriched job will include all necessary fields for display

### Code Changes
```javascript
// Remove: import data from '../data/mockData.json';
import { getEnrichedJob } from '../data/jobUtils';

// Replace: const JOBS = data.projects; (remove this line)

// Replace: const job = JOBS.find((item) => String(item.id) === jobId);
const job = getEnrichedJob(jobId);
```

## Bug 2: JobApply.jsx — job object accessed before null check
**Severity:** Critical  
**File:** `src/pages/JobApply.jsx`

### Current Issue
- Line 10: `const job = getEnrichedJob(jobId)` — may return `null`
- Lines 19-28: `formData` state initialization uses `job ?? {}` in `buildPortfolioApplicationIntro(profile, job ?? {}, portfolio)`
- Line 30: Null check `if (!job)` happens AFTER state initialization
- If `job` is null, the function receives an empty object, which may cause issues in `buildPortfolioApplicationIntro`

### Fix Plan
1. Move the null check to the very top of the component (before any state that references `job`)
2. Initialize state after confirming `job` exists
3. Use early return pattern for null case

## Bug 3: Jobs.jsx — Search state updates on every keystroke (Performance)
**Severity:** Medium  
**File:** `src/pages/Jobs.jsx`

### Current Issue
- Lines 177-180: `onChange` handler calls both `setSearchInput` and `setSearch` on every keystroke
- `setSearch` triggers recomputation of `filteredJobs` (lines 135-151) on every keystroke
- No debouncing — causes unnecessary re-renders with large job lists

### Fix Plan
1. Remove `setSearch` from the `onChange` handler
2. Only update search on form submit (already handled by `handleSearchSubmit`)
3. Keep `searchInput` for controlled input field
4. This matches the intended UX (click "Find Jobs" to filter) and improves performance

### Code Changes
```javascript
// Current (lines 177-180):
onChange={(event) => {
  setSearchInput(event.target.value);
  setSearch(event.target.value);  // REMOVE THIS LINE
}}

// New:
onChange={(event) => {
  setSearchInput(event.target.value);
}}
```

## Bug 4: Sidebar.jsx — Redundant storage event listener
**Severity:** Low  
**File:** `src/components/Sidebar.jsx`

### Current Issue
- Lines 24-27: Listens to `window.addEventListener('storage', refreshCounts)` 
- The `storage` event fires for ALL `localStorage` changes across all tabs/windows
- Custom events (`APPLICATIONS_CHANGED_EVENT`, `SAVED_JOBS_CHANGED_EVENT`) already handle intra-app synchronization
- This causes unnecessary re-renders when unrelated localStorage keys change

### Fix Plan
1. Remove the `storage` event listener (line 26)
2. Keep only the custom event listeners
3. Update cleanup function accordingly

### Code Changes
```javascript
// Remove this line:
window.addEventListener('storage', refreshCounts);

// And remove from cleanup:
window.removeEventListener('storage', refreshCounts);
```

### Current Issue
